# ملخص المحادثة - session-20260120t11084062323995

1) Vision
- منصة ويب (Projment) تربط أصحاب المشاريع (Founders) مع المستثمرين (Investors) والمستشارين (Advisors) والشركاء (Partners) عبر اشتراك مجاني في الـMVP، لعرض المشاريع والبحث عنها وإرسال العروض/الدعوات والتفاوض عبر دردشة نصية داخل المنصة، مع مراجعة إدارية قبل نشر المشاريع ودعم العربية والإنجليزية.

2) Personas
- Founder: إنشاء مشروع/مشاريع، إدخال بيانات المشروع، نشره لإرساله للمراجعة، استقبال/إرسال عروض ودعوات، التفاوض ثم القبول/الرفض.
- Investor: البحث والفلترة عن المشاريع “القابلة للنشر”، إرسال عروض استثمار/شراكة، التفاوض ثم القبول/الرفض.
- Advisor: عرض خبراته، إرسال/استقبال عروض استشارة، التفاوض ثم القبول/الرفض.
- Partner: عرض خبراته، إرسال/استقبال مقترحات شراكة/انضمام، التفاوض ثم القبول/الرفض.
- Admin: تحديد نوع حساب العميل (Founder/Investor/Advisor/Partner)، مراجعة المشاريع واعتمادها للنشر، إدارة المستخدمين والمحتوى والتقارير والإشعارات والبنرات والمعاملات.

4) Data / Content
- بيانات الحساب: البريد الإلكتروني، كلمة المرور، حالة التحقق، نوع الحساب (يحدده Admin).
- الملف الشخصي: الاسم، نبذة، الاهتمامات، المهارات، خبرات/مجالات، (وأي بيانات تعريفية أخرى ضمن الملف).
- بيانات المشروع (نموذج واحد): اسم المشروع (إلزامي)، شرح/وصف (إلزامي)، نوع المشروع (عام/خاص)، حالة المشروع (قيد المراجعة/قابل للنشر)، مرفقات PDF، وبيانات إضافية متفق عليها ضمن “البيانات المقترحة للمشروع”.
- بيانات العروض/الدعوات: المرسل/المستقبل، المشروع المرتبط (إن وجد)، نوع الطلب، الرسائل/سجل التفاوض، الحالة (مرسل/قيد التفاوض/مقبول/مرفوض).
- الدردشة: رسائل نصية، طابع زمني، أطراف المحادثة، ارتباطها بعرض/دعوة.
- الإشعارات: نوع الإشعار، المحتوى، حالة القراءة، قناة الإرسال (ويب/بريد).
- الإدارة: مستخدمون، مشاريع، عروض/دعوات، بانرات، تقارير.
- الدفع/المعاملات: المستخدم، التاريخ، المبلغ، وسيلة الدفع (بطاقة/Net Banking)، سجل العمليات.

5) Non-Functional Requirements
- الأداء: استجابة أقل من 500ms.
- الأمان: HTTPS، JWT، حماية من SQL Injection.
- التوافر: 99.5%.
- الخصوصية: لا يتم تخزين بيانات مالية أو عقود ملزمة.
- القابلية للتوسع: قابلية إضافة AI وWallet لاحقًا.
- دعم العربية والإنجليزية.
- الالتزام بسياسة حماية البيانات السعودية (PDPL).

الستاك (إلزامي)
- C# + ABP + Angular

6) Open / Missing Information
- تعريف دقيق لفروقات “عام/خاص” غير الإخفاء (تم حسم أن الخاص مخفي، لكن هل العام دائمًا ظاهر بعد الاعتماد؟).
- تفاصيل حقول المشروع الإضافية المعتمدة (غير الاسم والشرح وملفات PDF).
- تفاصيل حدود رفع ملفات PDF (عدد/حجم) بما أن الاشتراكات مجانية في الـMVP.
- تعريف “Trending”: معيار التفاعل (مشاهدات/عروض/رسائل/إعجابات…).
- قواعد البحث والفلترة المطلوبة (الحقول المتاحة للفلترة والترتيب).
- دورة حياة العرض/الدعوة وحالاتها التفصيلية (قيد التفاوض، من يغلق الطلب، إلخ).
- هل التفاوض قبل القبول سيكون داخل نفس الطلب فقط أم غرفة دردشة منفصلة مرتبطة به (لم يُحسم الشكل النهائي).
- تفاصيل الإشعارات المطلوبة (الأحداث التي ترسل بريد/ويب بالضبط).
- بوابة الدفع المختارة، مع أن الـMVP ذكر أن الخطط مجانية (تعارض يحتاج حسم: هل الدفع مؤجل أم موجود للترقية لاحقًا).
- سياسات إدارة المحتوى: أسباب الرفض في مراجعة المشاريع وهل تُعرض للمستخدم.
- صلاحيات Admin التفصيلية وتقارير الإدارة المطلوبة.
